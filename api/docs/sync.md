# Sync Rules V1

Первая версия синхронизации в «Иви» опирается на простой local-first сценарий.

## Базовая схема

- Android работает от локального `Room`
- сервер считается источником истины для общих данных питомца
- клиент пишет локальные изменения в `Room` и в outbox синхронизации
- синхронизация идет по циклу `push -> changes`
- при первом входе или потере курсора используется полный `bootstrap`

## Syncable сущности

- `UserProfile`
- `Pet`
- `PetMembership`
- `EventType`
- `PetEvent`
- `WeightEntry`

Примечание:

- серверная сущность `User` содержит `passwordHash`, но в публичный API и sync-поток отдается только безопасная модель `UserProfile`

Для syncable-моделей в API зафиксированы:

- стабильный UUID `id`
- серверная версия `version`
- время изменения `updatedAt`
- tombstone-признак через `deletedAt`

## Конфликты

- клиент отправляет `baseVersion` в каждой mutation
- сервер принимает mutation только если `baseVersion` совпадает с текущей версией записи
- при несовпадении сервер возвращает `409 Conflict` с `ConflictDto`
- в первой версии побеждает серверное состояние
- клиент должен обновить локальную копию и при необходимости предложить повторное ручное редактирование

## Что не синхронизируется в V1

- локальные `ReminderSettings`
- бинарные фото внутри JSON sync-потока

Фото питомца идут отдельным upload/delete контуром, а настройки напоминаний остаются локальными для устройства и UX.

## Что пока не входит в V1

- multi-pet
- сложные роли
- вложения у событий
- общие серверные reminder settings
- web-client
- сложный merge конфликтов
- realtime / websocket
- history / audit log
