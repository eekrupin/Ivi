# AGENTS.md

## Назначение документа
`AGENTS.md` — основной рабочий документ проекта «Иви».
Здесь фиксируются ключевые продуктовые и технические договоренности, границы текущего этапа, принятые допущения, архитектурные решения и ближайший план работ.
Если по проекту появляются новые важные решения, они сначала отражаются здесь.

## Текущее состояние репозитория
- В репозитории уже создан минимальный Android-проект на Gradle Kotlin DSL с одним модулем `:app`.
- Корневое имя проекта: `ivi`.
- Корневой package и `applicationId`: `ru.ekrupin.ivi`.
- Gradle wrapper добавлен в репозиторий: использовать `./gradlew`, а не системный `gradle`.
- `./gradlew :app:assembleDebug` и `./gradlew test` успешно проходят в рабочей среде.

## Цель проекта
«Иви» — Android-приложение для семейного использования, предназначенное для учета событий по одному питомцу, собаке по имени Иви.
Первая цель — получить качественный локальный MVP: удобно вести историю событий, веса и напоминаний, не усложняя проект раньше времени.
Архитектура должна позволять позже без боли добавить backend и синхронизацию между устройствами.

## Обязательные проектные договоренности
- Приложение называется «Иви» и делается как Android-приложение для личного семейного использования.
- Предметная область первого этапа: один питомец, локальный учет событий, история веса, локальные напоминания.
- Первый этап: только локальное Android-приложение. Не уходить в backend, sync, web, multi-pet и другую отложенную сложность без явной причины.
- Архитектуру закладывать с заделом под будущую синхронизацию, но без преждевременной реализации серверной части.

Что уже определено:
- платформа Android
- один модуль `:app`

- один питомец в рамках MVP

Что уже реализовано:

Что в работе:

Текущих технических блокеров среды не зафиксировано.

## Архитектурный подход
Базовый подход:
- один Android `app`-модуль на старте
- внутри четкое разделение на слои `presentation`, `domain`, `data`, `core`
- `Room` как единственный источник истины на первом этапе
- `Repository` как граница между доменом и источниками данных
- `ViewModel` на экран
- `Use case` только для сценариев с заметной бизнес-логикой, а не на каждую мелочь
- навигация через `Navigation Compose`
- отдельная абстракция для планирования уведомлений

Почему так:
- этого достаточно для локального MVP
- структура не перегружена
- позже можно добавить `remote` и sync-логику без слома UI и доменной модели

Справочник типов событий.

Поля:
- `id`
- `name`

- `colorArgb`
- `iconKey`
- `createdAt`
- `updatedAt`

Примеры:
- Бравекто 3
- Симпарика
- Прививка
- Глистогонное

### 4. Категория типа события
Для MVP достаточно enum или фиксированного справочника.

Базовые категории:
- От клещей
- От глистов
- Прививка
- Осмотр
- Другое

Решение:

### 5. Событие питомца
Основная рабочая сущность журнала.

Поля:
- `id`
- `petId`
- `eventTypeId`
- `eventDate`
- `dueDate`
- `comment`
- `notificationsEnabled`
- `status`
- `createdAt`
- `updatedAt`

Статусы:
- `ACTIVE`
- `COMPLETED`
- `ARCHIVED`

Правила:
- время не используется, только дата
- `dueDate` может быть рассчитана автоматически от `eventDate` + `defaultDurationDays`
- пользователь может вручную изменить `dueDate`
- события по умолчанию сортируются по убыванию `eventDate`
- в UI нужен режим просмотра как минимум для актуальных и архивных

### 6. Глобальные настройки напоминаний
Поля:
- `id`
- `firstReminderEnabled`
- `firstReminderDaysBefore`
- `secondReminderEnabled`
- `secondReminderDaysBefore`
- `createdAt`
- `updatedAt`

Правила:
- настройки едины для всего приложения
- напоминания локальные
- фактические даты напоминаний рассчитываются от `dueDate`
- если у события выключены уведомления, напоминания для него не создаются

## Ключевые бизнес-правила
- В приложении один питомец в рамках MVP.
- Имя питомца и фото редактируемые.
- Вес хранится только как история измерений.
- Текущий вес вычисляется, а не хранится отдельным обязательным полем.
- Типы событий нельзя физически удалять, если по ним уже есть события; их нужно деактивировать.
- Категории на первом этапе фиксированные.
- Событие всегда привязано к типу события.
- Дата события обязательна.
- Дата окончания или контроля необязательна, но чаще всего может быть рассчитана автоматически.
- Все пользовательские тексты в приложении только на русском языке.
- Пользователь не должен видеть технические имена полей и enum-значения.

## Текущее состояние реализации
- UI подключен к реальным данным из `Room` через `ViewModel` и локальные репозитории.
- Главный экран показывает питомца, текущий вес, ближайшее напоминание и список активных событий.
- Экран событий показывает реальные записи, умеет фильтровать по статусам и менять статус события.
- Экран типов событий показывает реальные типы и позволяет создавать и редактировать их локально.
- Экран веса показывает реальные записи и позволяет добавлять новые измерения.
- Экран настроек показывает и сохраняет реальные настройки напоминаний.
- Экран события работает как форма создания и редактирования с автоподстановкой даты контроля от типа события.
- Даты в формах выбираются через нативный Material 3 date picker.
- База стартово заполняется питомцем Иви, типами событий, несколькими событиями, несколькими записями веса и настройками напоминаний.
- `ReminderScheduler` больше не no-op: он полностью перестраивает расписание локальных уведомлений через `AlarmManager`.
- Для каждого подходящего события создается до двух локальных напоминаний в соответствии с глобальными настройками.
- При изменении события или глобальных настроек выполняется полное перепланирование: старые alarm-записи отменяются, затем создаются новые актуальные.
- После перезагрузки устройства, смены времени или часового пояса расписание восстанавливается через `ReminderRescheduleReceiver`.
- `DataStore` пока заявлен в стеке, но еще не подключен в коде.

## Напоминания
Выбранный механизм:
- `AlarmManager` с `setAndAllowWhileIdle` для локальных уведомлений на уровне Android
- `NotificationChannel` для канала уведомлений приложения
- `BroadcastReceiver` для показа уведомления
- отдельный `BroadcastReceiver` для полного перепланирования после `BOOT_COMPLETED`, `TIME_SET` и `TIMEZONE_CHANGED`

Как планируются уведомления:
- в расчет берутся только события с заполненной `dueDate`
- у события должны быть включены уведомления
- события в статусе `ARCHIVED` не планируются
- первое и второе напоминания вычисляются как `dueDate - N дней` по глобальным настройкам
- если конкретная дата напоминания уже в прошлом, такое напоминание не создается
- если оба напоминания в прошлом, для события ничего не планируется

Фиксированное время уведомления:
- для MVP выбрано фиксированное локальное время `10:00` утра по времени устройства
- отдельная пользовательская настройка времени уведомления пока не добавляется

Как работает перепланирование:
- после сохранения события вызывается полный `refreshAll()`
- после изменения глобальных настроек напоминаний вызывается полный `refreshAll()`
- `refreshAll()` сначала отменяет все ранее запланированные alarm-записи из локально сохраненного набора ключей, затем строит расписание заново по текущему состоянию БД

Что происходит после перезагрузки устройства:
- `ReminderRescheduleReceiver` получает системные broadcast-события после reboot и изменений времени
- после этого выполняется полное перепланирование по текущим данным `Room`

## Что сознательно не усложняем
- не делаем multi-module архитектуру по каждой фиче на старте
- не делаем отдельный backend-контур
- не строим sync engine заранее
- не вводим сложные generic base-классы
- не делаем отдельные use case на каждый простой запрос
- не добавляем ролевую модель и пользователей в локальную архитектуру
- не проектируем NAS как рабочее хранилище приложения

## Принятые допущения
- Пока поддерживается только один питомец.
- Фото питомца хранится локально, через URI или локальную ссылку на файл, без облачного слоя.
- Для дат в предметной области используется календарная дата без времени.
- Для Room и доменной модели даты лучше представлять через `LocalDate` с конвертерами.
- Категории типов событий на первом этапе фиксированные.
- Набор стартовых типов событий можно предзаполнить при первом запуске.
- Для ускорения локального MVP сидирование выполняется автоматически при пустой базе через `DatabaseSeeder`.
- Напоминания считаются от даты окончания или следующего контроля.
- Если дата контроля у события не задана, напоминания для него не планируются.
- Статус `COMPLETED` нужен для логического завершения сценария, `ARCHIVED` — для вывода из основного активного потока.
- Будущая синхронизация будет добавляться как отдельный `remote`-слой и механизм согласования изменений, а не как переработка всей модели.
- Для MVP напоминания ставятся на фиксированное локальное время `10:00`; отдельной настройки времени пока нет.
- Фото питомца пока хранится как строковый `photoUri`, без реального picker и загрузки изображения в UI.

## Открытые вопросы ближайшего этапа
- Точный механизм выбора и хранения фото питомца пока не зафиксирован; стартово ориентироваться на системный picker и локальный `Uri`.
- Точный стартовый список предзаполненных типов событий сейчас включает `Бравекто 3`, `Симпарика`, `Прививка`, `Глистогонное`; при необходимости его можно расширить.

## Команды
- Основная точка входа: `./gradlew`
- Проверить список задач: `./gradlew tasks`
- Собрать debug-сборку: `./gradlew :app:assembleDebug`
- Запустить unit-тесты: `./gradlew test`
- Запустить Android lint для debug: `./gradlew :app:lintDebug`

Порядок проверки после изменений:
1. `./gradlew :app:assembleDebug`
2. `./gradlew test`
3. `./gradlew :app:lintDebug`

Примечание:
- `./gradlew :app:assembleDebug` успешно проходит в рабочей среде
- при сборке есть предупреждение AGP `8.5.2` о `compileSdk 35`, но оно не блокирует сборку

## GitHub push
- Если пользователь явно просит `push`, перед пушем нужно проверить `git remote get-url origin` и `gh auth status`.
- Если `origin` указывает на `https://github.com/...` и `gh auth status` показывает активную сессию, не нужно сначала пробовать обычный `git push`, потому что в неинтерактивной среде он часто падает на запросе логина.
- В этом случае агент должен сразу использовать одноразовый push через временный URL вида `https://x-access-token:$(gh auth token)@github.com/<owner>/<repo>.git` без изменения `git remote` и `git config`.
- Такой push использовать только для текущей команды и только когда нужен реальный `push` по просьбе пользователя.

## Журнал ключевых решений

### D-001
Статус: принято
Решение:
- делаем локальный Android MVP без backend-реализации
Причина:
- сначала нужно получить полезное и устойчивое приложение, а не распыляться на инфраструктуру

### D-002
Статус: принято
Решение:
- используем Kotlin + Compose + Material 3 + Room
Причина:
- это современный и практичный стек для Android-приложения с локальным хранением

### D-003
Статус: принято
Решение:
- архитектура `presentation -> domain -> data`
Причина:
- этого достаточно для поддерживаемого MVP и будущего расширения под sync

### D-004
Статус: принято
Решение:
- один питомец в рамках MVP, но без захардкоженных данных в коде
Причина:
- это соответствует реальной задаче и не мешает масштабированию позже

### D-005
Статус: принято
Решение:
- категории типов событий делаем фиксированными на первом этапе
Причина:
- это упрощает модель и UI без потери пользы
