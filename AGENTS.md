# AGENTS.md

## Назначение документа
`AGENTS.md` — основной рабочий документ проекта «Иви».
Здесь фиксируются ключевые продуктовые и технические договоренности, границы текущего этапа, принятые допущения, архитектурные решения и ближайший план работ.
Если по проекту появляются новые важные решения, они сначала отражаются здесь.

## Текущее состояние репозитория
- Репозиторий уже переведен в переходный monorepo-вид: Android-клиент живет в отдельной папке `android/` как собственный Gradle root.
- Папка `api/` уже создана и содержит первую версию контракта backend API и синхронизации.
- Папка `backend/` уже создана как отдельный Gradle root backend-приложения на Kotlin + Ktor.
- Корневое имя проекта Android-сборки: `ivi`.
- Корневой package и `applicationId`: `ru.ekrupin.ivi`.
- Текущая рабочая точка входа Android-сборки из корня репозитория: `./android/gradlew -p ./android`.
- Текущая рабочая точка входа backend-сборки из корня репозитория: `./backend/gradlew -p ./backend`.
- Android-специфичные файлы Gradle wrapper, `gradle/`, `gradle.properties`, `local.properties`, `keystore.properties`, `keystore.properties.example` и модуль `app/` перенесены внутрь `android/`.
- Source of truth для API на текущем этапе: TypeSpec в `api/src/main.tsp`, OpenAPI генерируется как производный артефакт в `api/generated/openapi/openapi.yaml`.
- В backend уже подключены PostgreSQL, Flyway и Exposed DSL как база server-side persistence-слоя.
- Следующий архитектурный шаг: после backend skeleton отдельно добавить `infra/`, не смешивая это с уже зафиксированными Android-root, API-контрактом и серверным каркасом.

## Цель проекта
«Иви» — семейная система для учета событий по одному питомцу, собаке по имени Иви.
Первый этап уже дал локальный Android MVP: удобно вести историю событий, веса и напоминаний без лишней сложности.
Следующая цель — расширить проект до связки из Android-клиента и отдельного backend-приложения с синхронизацией между устройствами для двух пользователей, не ломая уже работающий локальный слой.

## Обязательные проектные договоренности
- Приложение называется «Иви» и делается как Android-приложение для личного семейного использования.
- Предметная область первого этапа: один питомец, локальный учет событий, история веса, локальные напоминания.
- Локальный Android MVP остается рабочей базой проекта; ломать его ради архитектурной красоты нельзя.
- Новый большой этап: отдельный backend на Kotlin + Ktor + PostgreSQL для общего доступа двух пользователей и синхронизации.
- Android и backend живут раздельно на уровне структуры репозитория и сборки; backend не встраивается внутрь Android `app`-модуля.
- Контракт между клиентом и сервером хранится отдельно в `api/` и не растворяется в коде одного из приложений.
- NAS Synology рассматривается только как будущий слой резервного копирования, а не как primary storage приложения.

## Текущий статус
Стадия проекта: локальный Android MVP стабилизирован, начинается архитектурный этап backend и синхронизации.

Что уже определено:
- платформа Android
- один модуль `:app` внутри отдельного Android root `android/`
- package `ru.ekrupin.ivi`
- приложение было локальным на первом этапе
- один питомец в рамках MVP
- будущая синхронизация нужна и теперь переходит из отложенной идеи в отдельный этап проекта
- backend будет отдельным Kotlin/Ktor-приложением
- основная серверная БД планируется на PostgreSQL
- серверная среда: Ubuntu mini server
- NAS не используется как primary storage

Что уже реализовано:
- `Room` подключен как реальное локальное хранилище
- заведены реальные DAO, entity, мапперы и локальные repository-реализации
- в `debug`-сборках база автоматически предзаполняется стартовыми данными при первом запуске; в `release` первый запуск должен быть пустым
- `home`, `events`, `eventedit`, `eventtypes`, `weight`, `settings` подключены к данным через `ViewModel`
- работает локальный CRUD для питомца, типов событий, событий, веса и настроек напоминаний
- для питомца работает реальный выбор фото через системный Android Photo Picker
- для питомца появился отдельный экран редактирования с именем, датой рождения, заменой и удалением фото
- работает фильтрация событий по статусам `ACTIVE`, `COMPLETED`, `ARCHIVED`
- работает физическое удаление событий из режима редактирования
- для типов событий из режима редактирования работает удаление с правилом: без связанных событий тип удаляется физически, при наличии связанных событий тип деактивируется
- формы используют нативный Material 3 date picker вместо ручного текстового ввода дат
- локальные напоминания подключены через `AlarmManager`, `BroadcastReceiver` и `NotificationChannel`
- перепланирование напоминаний работает после изменения событий, настроек и после перезагрузки устройства
- главный экран переработан в более живой обзорный экран с выразительной карточкой питомца, фото, возрастом, текущим весом, ближайшим важным блоком и быстрыми переходами
- создана папка `api/` с TypeSpec-контрактом первой версии для `auth`, `me`, `pets/current`, `invites`, `sync/bootstrap`, `sync/changes`, `sync/push`, `photo`, `health`
- OpenAPI 3.1 генерируется из TypeSpec в `api/generated/openapi/openapi.yaml`
- создана папка `backend/` с минимальным Ktor skeleton и отдельным Gradle root
- backend реально собирается и запускается через `./backend/gradlew -p ./backend`
- `GET /health` уже отвечает рабочим JSON-ответом
- route-группы `auth`, `me`, `pets/current`, `invites`, `sync`, `photo` заведены как server-side skeleton с stub-handler'ами под контракт
- backend реально подключается к PostgreSQL через JDBC/HikariCP
- Flyway автоматически применяет миграции при старте backend
- первая схема БД уже заведена миграцией `V1__foundation.sql` с таблицами `users`, `pets`, `pet_memberships`, `invites`
- вторая миграция `V2__refresh_tokens.sql` уже добавляет таблицу `refresh_tokens` для auth foundation
- persistence-слой для foundation-сущностей уже переведен с placeholder на реальный Exposed DSL repository-уровень без DAO
- `POST /v1/auth/register`, `POST /v1/auth/login`, `POST /v1/auth/refresh`, `GET /v1/me`, `POST /v1/pets`, `GET /v1/pets/current`, `POST /v1/pets/{petId}/invites`, `POST /v1/invites/accept` уже реализованы как реальные DB-backed handler-ы
- `GET /v1/sync/bootstrap` и `GET /v1/sync/changes` уже реализованы как реальные DB-backed sync-endpoint'ы
- `POST /v1/sync/push` уже реализован как реальный DB-backed mutation-endpoint для `event_types`, `pet_events`, `weight_entries`
- Android Room-модель уже расширена sync-метаданными для `pets`, `event_types`, `pet_events`, `weight_entries`, а также заведена локальная таблица `sync_outbox`
- локальные репозитории `event_types`, `pet_events`, `weight_entries` уже пишут мутации в outbox в одной Room-транзакции с записью локальных изменений
- Android-клиент уже получил первый client-side sync read foundation: network layer для `bootstrap`/`changes`, `sync_state`-хранилище cursor и транзакционный import/apply слой поверх Room
- Android-клиент уже получил первый client-side sync push foundation: drain `sync_outbox` через `POST /v1/sync/push` с обработкой `accepted`, `conflicts` и `requiresBootstrap`
- В приложении уже появился ручной app-level entrypoint sync V1 через экран настроек: полный flow связывает bootstrap / push / changes и показывает минимальный sync-статус
- В приложении уже появился foreground lifecycle-trigger sync V1: `MainActivity.onStart()` вызывает app-level runner поверх `RunFullSyncUseCase` с защитами от параллельных и слишком частых запусков
- client-side session/auth UX уже выведен из debug-only режима: backend URL, access token, refresh token и профиль пользователя теперь хранятся в отдельном DataStore-backed session store и используются foreground/manual/background sync
- В Android уже добавлен background sync foundation на WorkManager: периодический `SyncWorker` использует тот же `RunFullSyncUseCase`, что и ручной/foreground sync

Что в работе:
- подготовка простого серверного контура для двух пользователей и одного питомца
- подготовка следующего клиентского этапа после WorkManager foundation: conflict UX и/или дальнейшее усиление background orchestration
- подготовка `infra/` следующим этапом, уже без переноса Android и без переопределения API-контракта вручную
- локальная клиентская полировка больше не является единственным фокусом этапа

Текущих технических блокеров среды не зафиксировано.

## Архитектурный подход
Базовый подход:
- один Android `app`-модуль на старте
- внутри четкое разделение на слои `presentation`, `domain`, `data`, `core`
- `Room` как единственный источник истины на первом этапе
- `Repository` как граница между доменом и источниками данных
- `ViewModel` на экран
- `Use case` только для сценариев с заметной бизнес-логикой, а не на каждую мелочь
- навигация через `Navigation Compose`
- отдельная абстракция для планирования уведомлений
- на следующем этапе репозиторий становится monorepo с отдельными приложениями `android/` и `backend/`
- после появления аккаунтов и sync сервер становится источником истины для общих данных питомца, а Android остается local-first клиентом с `Room` как кэшем и рабочей офлайн-базой
- API-контракт и sync-модель живут отдельно от клиентского и серверного кода

Почему так:
- этого достаточно для локального MVP
- структура не перегружена
- позже можно добавить `remote` и sync-логику без слома UI и доменной модели
- отдельные приложения в одном репозитории проще сопровождать, чем пытаться прятать backend внутрь Android-сборки
- контракт в отдельной папке снижает связность и заставляет явно договориться о DTO, версиях и sync-правилах

## Планируемая структура проекта
Целевой вариант следующего этапа:

`android/`
- отдельный Gradle root Android-клиента
- внутри сохраняется текущий модуль `app` и его слои `app/`, `core/`, `data/`, `domain/`, `feature/`

`backend/`
- отдельный Gradle root backend-приложения на Kotlin + Ktor
- `src/main/kotlin/.../backend/` с `Application`, `config`, `routing`, `health`, `auth`, `me`, `pet`, `invite`, `sync`, `photo`, `db`, `common`
- `src/main/resources/` с `application.conf`, `logback.xml` и `db/migration/`
- отдельный wrapper и Gradle Kotlin DSL
- локальный `docker-compose.yml` для PostgreSQL-разработки

`api/`
- контракт между Android и backend
- TypeSpec-исходники как source of truth
- generated OpenAPI как производный артефакт
- описание DTO, sync-курсоров, конфликтов и примеры payload'ов

`infra/`
- `docker-compose` для локального PostgreSQL и сервисов окружения
- шаблоны `.env`
- deploy/dev-артефакты
- миграции и вспомогательные инфраструктурные скрипты

`AGENTS.md`
- основной рабочий документ проекта

Переходное правило:
- перенос Android Gradle root в `android/` выполняется отдельным структурным шагом без функциональных изменений и должен оставаться изолированным от backend-работ

Правило зависимостей:
- `presentation` знает `domain`
- `data` реализует контракты `domain`
- `domain` не зависит от Android UI и Room
- Android-специфичные детали уведомлений и БД изолированы в `data`
- Android не зависит напрямую от backend-реализации, а только от утвержденного API-контракта
- backend не зависит от Android-кода и не импортирует Android-модели как внутренний source of truth
- `api/` описывает внешний контракт, а не становится местом для смешивания runtime-логики двух приложений

Фактически уже заведено:
- `MainActivity` как entrypoint Activity
- `IviApplication` с `@HiltAndroidApp`
- `android/app/src/main/java/ru/ekrupin/ivi/app/ui/IviAppRoot.kt` как корневой Compose `Scaffold`
- `android/app/src/main/java/ru/ekrupin/ivi/app/navigation/IviNavGraph.kt` как корневой `NavHost`
- `android/app/src/main/java/ru/ekrupin/ivi/data/local/db/IviDatabase.kt` и базовые DAO/entity-классы
- `android/app/src/main/java/ru/ekrupin/ivi/data/local/seed/DatabaseSeeder.kt` для стартового наполнения базы в `debug`
- локальные repository-реализации в `android/app/src/main/java/ru/ekrupin/ivi/data/repository/`
- `ViewModel` для `home`, `events`, `eventedit`, `eventtypes`, `weight`, `settings`
- рабочие экраны для `home`, `events`, `eventedit`, `eventtypes`, `weight`, `settings`
- `android/app/src/main/java/ru/ekrupin/ivi/core/ui/DatePickerField.kt` как общий нативный компонент выбора даты
- `android/app/src/main/java/ru/ekrupin/ivi/data/reminder/ReminderPublisherReceiver.kt` для показа уведомлений

## Предметная модель

### 1. Питомец
Сущность одного питомца.

Поля:
- `id`
- `name`
- `birthDate`
- `photoUri`
- `createdAt`
- `updatedAt`

Примечание:
- приложение пока для одного питомца, но имя и данные не должны быть захардкожены

### 2. Запись веса
История измерений веса.

Поля:
- `id`
- `petId`
- `date`
- `weightGrams`
- `comment`
- `createdAt`

Правило:
- текущий вес определяется по последней актуальной записи

Примечание:
- в коде сейчас используется имя `WeightEntry`; это рабочая реализация сущности записей веса для MVP

### 3. Тип события
Справочник типов событий.

Поля:
- `id`
- `name`
- `category`
- `defaultDurationDays`
- `isActive`
- `colorArgb`
- `iconKey`
- `createdAt`
- `updatedAt`

Примеры:
- Бравекто 3
- Симпарика
- Прививка
- Глистогонное
- Осмотр

### 4. Категория типа события
Для MVP достаточно enum или фиксированного справочника.

Базовые категории:
- От клещей
- От глистов
- Прививка
- Осмотр
- Другое

Решение:
- на первом этапе использовать enum, чтобы не раздувать модель без необходимости

### 5. Событие питомца
Основная рабочая сущность журнала.

Поля:
- `id`
- `petId`
- `eventTypeId`
- `eventDate`
- `dueDate`
- `comment`
- `notificationsEnabled`
- `status`
- `createdAt`
- `updatedAt`

Статусы:
- `ACTIVE`
- `COMPLETED`
- `ARCHIVED`

Правила:
- время не используется, только дата
- `dueDate` может быть рассчитана автоматически от `eventDate` + `defaultDurationDays`
- пользователь может вручную изменить `dueDate`
- события по умолчанию сортируются по убыванию `eventDate`
- в UI нужен режим просмотра как минимум для актуальных и архивных
- напоминания планируются только для событий в статусе `ACTIVE`
- статус `COMPLETED` означает, что сценарий завершён и новые напоминания для него больше не планируются
- статус `ARCHIVED` означает вывод события из обычного сценария и отсутствие планирования напоминаний

### 6. Глобальные настройки напоминаний
Поля:
- `id`
- `firstReminderEnabled`
- `firstReminderDaysBefore`
- `secondReminderEnabled`
- `secondReminderDaysBefore`
- `createdAt`
- `updatedAt`

Правила:
- настройки едины для всего приложения
- напоминания локальные
- фактические даты напоминаний рассчитываются от `dueDate`
- если у события выключены уведомления, напоминания для него не создаются

## Ключевые бизнес-правила
- В приложении один питомец в рамках MVP.
- Имя питомца и фото редактируемые.
- Вес хранится только как история измерений.
- Текущий вес вычисляется, а не хранится отдельным обязательным полем.
- Типы событий нельзя физически удалять, если по ним уже есть события; их нужно деактивировать.
- Из UI для редактирования типа события доступна кнопка удаления: если связанных событий нет, тип удаляется физически; если связанные события есть, вместо удаления тип деактивируется.
- Категории на первом этапе фиксированные.
- Событие всегда привязано к типу события.
- Дата события обязательна.
- Дата окончания или контроля необязательна, но чаще всего может быть рассчитана автоматически.
- При создании нового события статус в форме не показывается и новое событие всегда сохраняется как `ACTIVE`.
- Напоминания планируются только для событий со статусом `ACTIVE`.
- Для статусов `COMPLETED` и `ARCHIVED` напоминания не планируются.
- Все пользовательские тексты в приложении только на русском языке.
- Пользователь не должен видеть технические имена полей и enum-значения.

## Правила именования
В коде:
- технические имена на английском
- классы и файлы по общим Kotlin-правилам
- названия сущностей согласованы с предметной моделью: `Pet`, `WeightEntry`, `EventType`, `PetEvent`, `ReminderSettings`

В UI:
- только русский язык
- никаких техназваний вида `eventType`, `notificationEnabled`, `dueDate`
- все подписи, сообщения, пустые состояния и кнопки формулируются человеческим языком

## Правила локализации UI
- Все строки выносятся в ресурсы.
- Базовая локаль интерфейса — русский язык.
- Даже если внутренняя модель использует английские ключи и enum, пользователь видит только русские названия.
- Формулировки должны быть короткими, понятными и неканцелярскими.

## UI-подход для MVP

### Главный экран
Не просто список, а обзор текущего состояния:
- карточка питомца с фото
- понятный placeholder, если фото еще не выбрано
- действие для выбора или замены фото прямо из карточки
- текущий вес
- ближайшее важное напоминание
- блок актуальных событий
- быстрые действия: добавить событие, открыть события, открыть вес, открыть настройки
- редактирование данных питомца вынесено в отдельный экран, чтобы `home` оставался обзорным

### События
- список карточек
- фильтры по статусам
- сортировка по убыванию даты
- понятные маркеры типа события и статуса

### Форма события
- короткая и понятная форма
- выбор типа события
- дата события
- автоматический расчет даты контроля с возможностью ручной правки
- комментарий
- переключатель уведомлений

- при редактировании существующего события статус можно менять и можно удалить событие физически
- блоки формы сгруппированы по смыслу: основные данные, статус и уведомления

### Вес
- текущий вес крупно
- история записей

- два глобальных правила напоминаний
- простое объяснение, за сколько дней придет уведомление

## Текущее состояние реализации
- UI подключен к реальным данным из `Room` через `ViewModel` и локальные репозитории.
- Главный экран показывает питомца, фото, возраст, текущий вес, ближайший важный блок и список активных событий.
- На главном экране есть дружелюбные пустые состояния для фото, важных блоков и активных событий.
- Экран событий показывает реальные записи, умеет фильтровать по статусам и менять статус события.
- Экран типов событий показывает реальные типы и позволяет создавать и редактировать их локально.
- Экран веса показывает реальные записи, позволяет добавлять новые измерения и имеет аккуратное пустое состояние.
- Экран настроек показывает и сохраняет реальные настройки напоминаний.
- Экран события работает как форма создания и редактирования с автоподстановкой даты контроля от типа события.
- При создании события статус не показывается и новая запись всегда стартует как `ACTIVE`.
- В редактировании события доступно физическое удаление.
- У питомца есть отдельный экран редактирования, где можно менять имя, дату рождения, выбирать, заменять и удалять фото.
- Основные экраны прошли финальный локальный полировочный этап: выровнены карточки, пустые состояния, вторичные тексты, action-кнопки и визуальный ритм.
- Список событий стал чище: статус читается отдельным акцентом, а карточки лучше различаются по состояниям `ACTIVE`, `COMPLETED` и `ARCHIVED`.
- Форма события стала яснее за счет смысловой группировки блоков и более спокойных пояснений по статусу и напоминаниям.
- Экран настроек уведомлений и экран веса приведены к более цельному и менее техническому виду.
- Даты в формах выбираются через нативный Material 3 date picker.
- Фото питомца выбирается через системный Android Photo Picker и сохраняется как локальная копия в приватном хранилище приложения.
- В поле `photoUri` хранится строковый `file://` URI на локальную копию изображения внутри файлов приложения.
- В `debug`-сборках база стартово заполняется питомцем Иви, типами событий, несколькими событиями, несколькими записями веса и настройками напоминаний.
- `ReminderScheduler` больше не no-op: он полностью перестраивает расписание локальных уведомлений через `AlarmManager`.
- Для каждого подходящего события создается до двух локальных напоминаний в соответствии с глобальными настройками.
- При изменении события или глобальных настроек выполняется полное перепланирование: старые alarm-записи отменяются, затем создаются новые актуальные.
- После перезагрузки устройства, смены времени или часового пояса расписание восстанавливается через `ReminderRescheduleReceiver`.
- Разрешение на уведомления не запрашивается автоматически при старте; пользователь управляет этим сценарием из экрана настроек напоминаний.
- Форма события теперь явно показывает статус и объясняет влияние статуса на напоминания.
- `DataStore` пока заявлен в стеке, но еще не подключен в коде.

## Напоминания
Выбранный механизм:
- `AlarmManager` с `setAndAllowWhileIdle` для локальных уведомлений на уровне Android
- `NotificationChannel` для канала уведомлений приложения
- `BroadcastReceiver` для показа уведомления
- отдельный `BroadcastReceiver` для полного перепланирования после `BOOT_COMPLETED`, `TIME_SET` и `TIMEZONE_CHANGED`

Как планируются уведомления:
- в расчет берутся только события с заполненной `dueDate`
- у события должны быть включены уведомления
- планируются только события в статусе `ACTIVE`
- события в статусах `COMPLETED` и `ARCHIVED` не планируются
- первое и второе напоминания вычисляются как `dueDate - N дней` по глобальным настройкам
- если конкретная дата напоминания уже в прошлом, такое напоминание не создается
- если оба напоминания в прошлом, для события ничего не планируется

Фиксированное время уведомления:
- для MVP выбрано фиксированное локальное время `10:00` утра по времени устройства
- отдельная пользовательская настройка времени уведомления пока не добавляется

Как работает перепланирование:
- после сохранения события вызывается полный `refreshAll()`
- после изменения глобальных настроек напоминаний вызывается полный `refreshAll()`
- `refreshAll()` сначала отменяет все ранее запланированные alarm-записи из локально сохраненного набора ключей, затем строит расписание заново по текущему состоянию БД

Что происходит после перезагрузки устройства:
- `ReminderRescheduleReceiver` получает системные broadcast-события после reboot и изменений времени
- после этого выполняется полное перепланирование по текущим данным `Room`

UX разрешений на уведомления:
- системный запрос на уведомления не показывается агрессивно при первом старте приложения
- экран настроек показывает, доступны ли уведомления на уровне системы и приложения
- если уведомления недоступны, пользователь видит короткое объяснение, зачем они нужны
- из настроек можно запросить разрешение `POST_NOTIFICATIONS` или открыть системные настройки приложения
- при отказе приложение продолжает работать, но честно показывает, что напоминания не будут отображаться

## Следующий большой этап: backend и синхронизация

### Целевая модель репозитория
- Репозиторий становится monorepo из двух отдельных приложений: Android-клиента и backend-сервера.
- Android после выделения должен жить в `android/`, backend — в `backend/`.
- На первом переходном шаге допустимо оставить текущий Android root на месте и сначала утвердить контракт, а физический перенос выполнить отдельным коммитом.
- `api/` должен хранить версионируемый контракт: TypeSpec-исходники, generated OpenAPI, DTO, sync-правила и примеры запросов/ответов.
- `infra/` должен содержать только окружение и инфраструктурные артефакты, а не бизнес-логику.

Правила remote-модели:
- Для syncable-сущностей использовать стабильные UUID на сервере и в клиентском слое синхронизации.
- У каждой syncable-записи должны быть `updatedAt`, `version` и `deletedAt` для курсорной синхронизации и tombstone-удалений.
- `ReminderSettings` на первом sync-этапе не делать общей серверной сущностью: это локальная или пользовательская настройка показа уведомлений, а не общий источник истины для данных питомца.
- Фото питомца синхронизировать отдельным бинарным API, а не встраивать в общий JSON sync-поток.
- Текущий Android `photoUri` остается локальным указателем на кэш или локальную копию; для сервера нужен отдельный идентификатор ревизии фото.

### Простая модель доступа для двух пользователей и одного питомца
- На первом серверном этапе поддерживается один общий питомец на один совместный контур данных.
- Доступ дают два пользователя максимум в рамках одного питомца.
- Базовые роли: `OWNER` и `MEMBER`.
- `OWNER` создает питомца, приглашает второго пользователя и при необходимости может отозвать доступ.
- `MEMBER` имеет полный доступ на чтение и изменение данных питомца, типов событий, событий и веса.
- Сложную ролевую модель, семьи, команды и произвольное число участников пока не вводим.
- Для первого шага достаточно схемы: регистрация пользователя, создание питомца, генерация invite-кода или invite-token, принятие приглашения вторым пользователем.
- Практическое ограничение текущей V1-реализации: пользователь не может одновременно быть привязан к нескольким питомцам; если у пользователя уже есть активный membership, создать нового питомца или принять приглашение к другому питомцу нельзя.

### Стратегия синхронизации Android Room ↔ backend
- Android остается local-first приложением: UI работает от `Room`, а не напрямую от сетевых DTO.
- Для sync в Android позже понадобится отдельный слой `data/remote` и `data/sync`, не смешанный с текущими локальными DAO.
- Локальные изменения должны записываться в `Room` и в outbox синхронизации в одной транзакции.
- Для каждой syncable-сущности в локальной базе понадобятся sync-метаданные или отдельная таблица соответствий: `remoteId`, `serverVersion`, `syncState`, `lastSyncedAt`.
- Удаления лучше передавать через outbox/tombstone-модель, а не полагаться только на физическое удаление локальной строки.
- Первая инициализация после входа пользователя: `bootstrap` со снимком всех доступных данных питомца.
- Текущий `bootstrap` возвращает текущего пользователя, текущего питомца, все активные membership текущего питомца, а также `event_types`, `pet_events` и `weight_entries` этого питомца.
- Для sync-consistency `bootstrap` уже включает записи с заполненным `deletedAt`, чтобы клиент мог корректно восстановиться после потери курсора и не resurrect-ить локально уже удаленные на сервере объекты.
- `changes` работает по модели `(cursor, highWatermark]`: сервер декодирует cursor во внутренний timestamp, выбирает новый `highWatermark = now()` и возвращает только записи с `updatedAt > cursorTime` и `updatedAt <= highWatermark`.
- В `changes` обычные измененные записи возвращаются в секции `changes.*`, а soft-deleted записи — в `tombstones`; это согласовано с тем, что `bootstrap` отдает полный срез вместе с soft-deleted объектами.
- Для `changes` сервер принимает оба известных формата opaque cursor: `bootstrap:<epochMillis>` и `changes:<epochMillis>`; в ответе `changes` всегда выдается новый cursor вида `changes:<epochMillis>`.
- Для изменений membership сервер пока возвращает только активные membership, потому что revoke/delete сценарий для membership еще не реализован на persistence/API уровне V1.
- Рабочий цикл синхронизации: push локального outbox, затем pull изменений по курсору, затем перепланирование локальных напоминаний по уже синхронизированному состоянию `Room`.
- При потере курсора или несовместимости версии протокола допустим fallback на полный `bootstrap`, а не сложное восстановление частичных дельт.
- Невалидный cursor в `changes` приводит к `409 invalid_sync_cursor`, а отсутствие параметра `cursor` — к `400 missing_sync_cursor`; в обоих случаях клиент может сделать fallback на полный `bootstrap`.
- Отдельного серверного TTL или expiration для cursor в текущей V1-модели пока нет: если формат курсора валиден, сервер пытается обработать его как timestamp-high-watermark.

### Простое разрешение конфликтов на первом этапе
- Конфликтом считаем только ситуацию, когда один и тот же объект был изменен на двух клиентах от одной и той же базовой серверной версии.
- Разные записи сливаются независимо и автоматически.
- Для одной записи используем optimistic concurrency по `version`.
- Сервер принимает изменение только если `baseVersion` клиента совпадает с текущей серверной версией записи.
- При несовпадении сервер возвращает конфликт с актуальной серверной записью.
- На первом этапе серверная версия считается победившей, клиент обновляет локальные данные из ответа сервера и помечает запись как требующую ручного повторного редактирования, если пользователь хочет снова применить свои правки.
- Такой подход проще и честнее, чем неявный last-write-wins по сомнительным клиентским часам.

### Верхнеуровневый API-контракт
Минимальный стартовый набор API:

- Auth
  - `POST /v1/auth/register`
  - `POST /v1/auth/login`
  - `POST /v1/auth/refresh`
  - `GET /v1/me`

- Pet access
  - `POST /v1/pets`
  - `GET /v1/pets/current`
  - `POST /v1/pets/{petId}/invites`
  - `POST /v1/invites/accept`

- Sync
  - `GET /v1/sync/bootstrap`
  - `GET /v1/sync/changes?cursor=...`
  - `POST /v1/sync/push`

- Photo
  - `PUT /v1/pets/{petId}/photo`
  - `DELETE /v1/pets/{petId}/photo`

- Service
  - `GET /health`

Правила API-контракта:
- Главный мобильный контракт первого sync-этапа строится вокруг `bootstrap`, `changes`, `push`, а не вокруг россыпи отдельных CRUD-эндпоинтов на каждый экран.
- Source of truth для API — TypeSpec в `api/src/main.tsp`; OpenAPI редактировать вручную нельзя, он должен генерироваться из TypeSpec.
- `api/` должен фиксировать DTO явно: `SyncBootstrapResponse`, `SyncChangesResponse`, `SyncPushRequest`, `SyncPushResponse`, `ConflictDto`, `InviteDto`, `AuthTokenPairDto`.
- `SyncPushRequest` должен содержать `deviceId`, последнюю известную позицию курсора и список мутаций с `entityType`, `operation`, `baseVersion` и payload.
- `SyncChangesResponse` должен содержать новый `cursor`, набор измененных сущностей, tombstone-удаления и признак, нужен ли повторный запрос следующей страницы изменений.
- В первой версии TypeSpec-контракта уже зафиксированы группы endpoint `auth`, `me`, `pets/current`, `invites`, `sync/bootstrap`, `sync/changes`, `sync/push`, `photo`, `health`.
- `ReminderSettings` остаются локальными и не входят в первую server-sync схему; фото питомца идут отдельным binary-контуром, а не в JSON sync-потоке.

### Auth foundation первого этапа
- Модель входа: `email + password`.
- Пароль хранится только в виде `BCrypt`-хэша в таблице `users.password_hash`.
- Для доступа используется короткоживущий `access token` в виде JWT `Bearer`-токена с `userId` в `sub`.
- Для продления сессии используется отдельный `refresh token` как opaque random token.
- `refresh token` хранится на сервере только в виде SHA-256 hash в таблице `refresh_tokens`, а не в открытом виде.
- При `refresh` старый refresh token ревокается и выдается новая пара `access + refresh`.
- Публичные endpoint текущего этапа: `register`, `login`, `refresh`, `health`.
- Требуют авторизации: `me`, `pets`, `invites`.

### Invite flow первого этапа
- Приглашение создает только пользователь с ролью `OWNER` для соответствующего питомца.
- При создании генерируется простой opaque invite code с TTL в часах.
- В текущей реализации TTL приглашения ограничен диапазоном от 1 до 168 часов; по умолчанию используется 72 часа.
- Принять приглашение может только авторизованный пользователь.
- После принятия приглашения создается `MEMBER`-membership к тому же питомцу, а само приглашение переводится в `ACCEPTED`.
- Повторное использование, истечение срока или отсутствие приглашения возвращают явную ошибку.

### Аккуратный переход от текущей структуры к новой
- Перенос Android в `android/` уже выполнен как отдельный структурный шаг без функциональных изменений.
- Android Gradle root, wrapper-файлы, `gradle/`, `app/`, `local.properties` и release-signing файлы теперь живут внутри `android/`.
- После такого переноса backend-каркас и API-контракт можно добавлять отдельно, не трогая клиентскую структуру.
- При работе с release signing использовать Android root как базовую директорию для `keystore.properties` и связанных путей.
- `api/` и `infra/` можно создавать рано, потому что они не ломают текущий клиент и помогают согласовать новый этап.
- Backend сейчас обязан реализовывать контракт из `api/`, а не формировать собственный параллельный HTTP-контракт; ориентир для реализации — TypeSpec source и сгенерированный `api/generated/openapi/openapi.yaml`.
- На текущем этапе persistence в backend строится на PostgreSQL + Flyway + Exposed DSL; Exposed DAO сознательно не используем.

## Что сознательно не усложняем
- не делаем multi-module архитектуру по каждой фиче на старте
- не делаем несколько backend-сервисов; на первом этапе нужен один Ktor backend
- не строим event sourcing, CRDT, per-field merge и другую тяжелую sync-математику
- не вводим сложные generic base-классы
- не делаем отдельные use case на каждый простой запрос
- не вводим роли сложнее `OWNER` и `MEMBER`
- не добавляем multi-pet в первый серверный этап
- не проектируем NAS как рабочее хранилище приложения
- не используем Exposed DAO; достаточно DSL-стиля и явных repository

## Принятые допущения
- Пока поддерживается только один питомец.
- Фото питомца хранится локально, без облачного слоя.
- Для дат в предметной области используется календарная дата без времени.
- Для Room и доменной модели даты лучше представлять через `LocalDate` с конвертерами.
- Категории типов событий на первом этапе фиксированные.
- Набор стартовых типов событий можно предзаполнять при первом запуске только в `debug`-сборках.
- Для ускорения локального MVP сидирование выполняется автоматически при пустой базе через `DatabaseSeeder` только в `debug`; `release` должен стартовать без демо-данных.
- Напоминания считаются от даты окончания или следующего контроля.
- Если дата контроля у события не задана, напоминания для него не планируются.
- Статус `COMPLETED` нужен для логического завершения сценария, `ARCHIVED` — для вывода из основного активного потока.
- Статус `COMPLETED` исключает событие из дальнейшего планирования напоминаний так же, как и `ARCHIVED`.
- Будущая синхронизация будет добавляться как отдельный `remote`-слой и механизм согласования изменений, а не как переработка всей модели.
- Для MVP напоминания ставятся на фиксированное локальное время `10:00`; отдельной настройки времени пока нет.
- Для MVP выбран системный Android Photo Picker без собственной галереи и без множественного выбора.
- После выбора фото копируется во внутреннее хранилище приложения, поэтому отдельное долгоживущее permission на внешний URI не требуется.
- В `photoUri` сохраняется URI локальной копии файла в приватной директории приложения; текущая реализация ориентирована только на отображение фото внутри приложения.
- При удалении фото из экрана редактирования локальный файл тоже удаляется, если он находится в управляемой директории приложения.
- В следующем этапе backend разворачивается как отдельное приложение на Ubuntu mini server.
- Основное серверное хранилище — PostgreSQL.
- Synology NAS остается будущим слоем резервного копирования и не участвует в online-записях как primary storage.
- Для серверной синхронизации каждому общему объекту нужен стабильный серверный UUID и версия записи.
- После появления аккаунтов и sync сервер становится источником истины для общих данных питомца, но Android сохраняет local-first UX через `Room`.
- Глобальные настройки локальных уведомлений не синхронизируются между двумя пользователями как общий объект первого этапа.
- Фото питомца должно уйти в отдельный upload/download-контур и не должно опираться на текущие `file://` URI за пределами одного устройства.

## Открытые вопросы ближайшего этапа
- Нужна ли синхронизация фото питомца в первой серверной поставке или ее разумнее вынести во второй backend-итерации после запуска структурированных данных.
- Нужен ли уже в первой версии backend отдельный refresh-token flow, или на старте можно держать только короткоживущий access token + явный login.

## GitHub push
- Если пользователь явно просит `push`, перед пушем нужно проверить `git remote get-url origin` и `gh auth status`.
- Если `origin` указывает на `https://github.com/...` и `gh auth status` показывает активную сессию, не нужно сначала пробовать обычный `git push`, потому что в неинтерактивной среде он часто падает на запросе логина.
- В этом случае агент должен сразу использовать одноразовый push через временный URL вида `https://x-access-token:$(gh auth token)@github.com/<owner>/<repo>.git` без изменения `git remote` и `git config`.
- Такой push использовать только для текущей команды и только когда нужен реальный `push` по просьбе пользователя.

### Клиентский read-path sync
- На текущем шаге Android получил минимальный network layer для `GET /v1/sync/bootstrap` и `GET /v1/sync/changes` через отдельный remote data source.
- Текущий sync cursor и время последних успешных read-операций хранятся локально в Room-таблице `sync_state`.
- Для V1 выбрано простое правило bootstrap import: он разрешен только при пустом `sync_outbox`, потому что локальные несинхронизированные мутации и серверный snapshot пока не пытаются автоматически merge-иться на клиенте.
- Bootstrap import сейчас считается authoritative replace для syncable server-backed данных питомца: клиент транзакционно обновляет `pets`, полностью заменяет локальные `event_types`, `pet_events`, `weight_entries`, а также обновляет локальный cache пользователей и membership.
- При bootstrap import локальное `photoUri` питомца сохраняется, потому что фото пока не входит в server-sync контур.
- Changes pull берет сохраненный cursor из `sync_state`, запрашивает incremental changes и применяет обычные изменения и tombstones поверх локальной базы без очистки таблиц.
- И bootstrap import, и changes pull после успешного применения обновляют `syncState = SYNCED`, `serverVersion`, `serverUpdatedAt`, `deletedAt`, `lastSyncedAt` и сохраняют новый cursor.
- Текущая клиентская sync-реализация пока не вызывает `sync/push`, не дренирует outbox и не запускается автоматически из UI/background — это будет следующим шагом.

### Клиентский push path
- На текущем шаге Android получил client-side вызов `POST /v1/sync/push` через уже существующий sync network layer.
- Drain outbox берет `PENDING` записи из `sync_outbox`, помечает их как `IN_FLIGHT`, формирует пакет `mutations` и отправляет его на сервер.
- Для V1 выбрана простая terminal-стратегия для успешно принятых мутаций: после обработки `accepted` соответствующие outbox-записи физически удаляются.
- Для `accepted` клиент обновляет локальные сущности: `serverVersion`, `serverUpdatedAt`, `syncState = SYNCED`, `lastSyncedAt`.
- Для `conflicts` выбран простой и честный V1-подход: серверный `serverRecord` пока не применяется автоматически поверх локальной сущности; локальная запись переводится в `syncState = CONFLICT`, а соответствующая outbox-запись помечается как `FAILED`.
- Для `requiresBootstrap = true` клиент не считает push успешным: текущий пакет помечается `FAILED`, а в `sync_state.requiresBootstrap = true` фиксируется необходимость следующего полного bootstrap.
- После успешного `push` клиент сохраняет cursor из server-side ответа как новый read-cursor; отдельный автоматический follow-up `changes` после push в текущей V1-реализации пока не выполняется.
- Первый end-to-end cycle теперь выглядит так: локальная мутация -> запись в `sync_outbox` -> drain `sync/push` -> `accepted/conflicts` -> обновление локальной сущности и `sync_state`.

### App-level sync orchestration V1
- В приложении теперь есть отдельный `RunFullSyncUseCase`, который объединяет уже существующие bootstrap import, changes pull и drain outbox.
- Для V1 выбран простой app-level порядок: если `requiresBootstrap = true` или cursor отсутствует, то при пустом outbox выполняется `bootstrap`; если cursor уже есть, то при наличии pending outbox сначала выполняется `push`, затем `changes`.
- Если нужен bootstrap, но outbox уже не пустой, orchestration не пытается автоматически merge-ить состояния и возвращает отдельный результат `RequiresBootstrap`.
- Если `push` завершился конфликтами, общий sync cycle не считается полностью успешным: orchestration возвращает `ConflictsDetected`, но всё равно выполняет `changes` для выравнивания server-driven состояния.
- Ручной запуск sync сейчас доступен из экрана настроек через секцию синхронизации: пользователь может сохранить конфигурацию backend, сбросить её и вручную запустить sync.
- На уровне приложения пока показываются минимальные статусы: `NotConfigured`, `Running`, `Success`, `ForegroundSuccess`, `Conflicts`, `RequiresBootstrap`, `Error`.
- Session/config модель теперь отделена от operational sync state: конфигурация sync хранится в отдельном DataStore-backed `SyncConfigStore`, а `sync_state` используется только для cursor, флагов и runtime-состояния sync.

### Session/config UX V1
- Для V1 выбран один источник client-side session/config: `SyncSessionStore` на основе Android DataStore Preferences.
- В `SyncSessionStore` хранятся `baseUrl`, `accessToken`, `refreshToken`, а также минимальный профиль пользователя (`userId`, `email`, `displayName`).
- При первом чтении `SyncSessionStore` выполняет одноразовую мягкую миграцию legacy-конфигурации из `sync_state`, если она там еще есть, после чего foreground/manual/background sync читают уже только session store.
- `AppSyncRunner`, `SyncWorker` и ручной запуск в настройках используют один и тот же `SyncSessionStore`; прямой зависимости на хранение session/config в `sync_state` больше нет.
- Экран настроек теперь выступает как минимальный экран подключения: пользователь вводит backend URL, email, пароль и может войти или зарегистрироваться через уже существующий backend auth.
- После успешного login/register локальная сессия сохраняется в `SyncSessionStore`, и foreground/manual/background sync начинают использовать её автоматически.
- Logout/reset connection очищает session store целиком; после этого foreground и background sync корректно молчат, пока пользователь не войдет заново.
- Текущее V1-решение сознательно не является полным production-grade auth UX: отдельного экрана логина нет, refresh выполняется скрыто для sync, а более богатый session UX и polishing откладываются на следующий этап.

### Foreground sync trigger V1
- На текущем шаге автоматический foreground-trigger привязан к `MainActivity.onStart()`: при старте приложения и каждом возврате activity в foreground вызывается `AppSyncRunner.triggerForegroundSync()`.
- Foreground-trigger не содержит собственной sync-логики и использует уже существующий `RunFullSyncUseCase` через общий app-level runner.
- Для V1 добавлены простые защитные правила: если sync уже выполняется, новый запуск не стартует; если backend session не сохранена, auto-sync не запускается; если предыдущий foreground sync стартовал менее 30 секунд назад, новый запуск пропускается.
- Foreground-trigger обновляет тот же app-level sync status, что и ручной запуск, поэтому текущее состояние можно увидеть в экране настроек без отдельного UI.
- Ошибки foreground sync пока не показываются агрессивно пользователю: они отражаются только в текущем sync-статусе, без popup/snackbar-шума.
- Текущий foreground-trigger работает только пока жива `MainActivity`; это сознательно не background sync и не замена будущей WorkManager/background orchestration модели.

### Background sync foundation V1
- На текущем шаге в Android добавлен один `SyncWorker` на базе WorkManager; worker использует уже существующий `RunFullSyncUseCase` и не содержит отдельной sync-логики.
- Scheduling строится через `BackgroundSyncScheduler`, который при старте приложения регистрирует unique periodic work `ivi-background-sync` с policy `KEEP`.
- Для V1 выбрана простая периодическая стратегия: один periodic worker с network constraint `CONNECTED`, без отдельного one-shot scheduling по запросу.
- Background worker использует тот же `SyncSessionStore`, что и ручной и foreground sync; если sync не настроен или пользователь не вошел, worker завершается `success` без шума.
- Для предотвращения параллельных запусков foreground/manual/background sync используют общий `SyncExecutionGate`; если другой sync уже идет, worker тихо завершается `success`.
- Для V1 worker трактует результаты так: `Success`, `ConflictsDetected`, `RequiresBootstrap`, `AuthError`, `ValidationError` -> `Result.success()`; `NetworkError`, `ServerError` c 5xx и `UnknownError` -> `Result.retry()`.
- Background sync сейчас intentionally quiet: без пользовательских popup, без системных уведомлений, без conflict UI.

### D-023
Статус: принято
Решение:
- первый client-side sync read path строится вокруг отдельного remote data source, Room-таблицы `sync_state` и транзакционного import/apply слоя; bootstrap в V1 разрешен только при пустом outbox и выполняет authoritative replace server-backed данных
Причина:
- это дает работающую и понятную read-side интеграцию с уже готовым server-side sync trio, не пытаясь преждевременно решить сложный merge локальных несинхронизированных изменений с серверным snapshot

### D-024
Статус: принято
Решение:
- первый client-side push path дренирует `sync_outbox` пакетами `PENDING -> IN_FLIGHT`, удаляет `accepted` записи, переводит конфликтующие записи в `FAILED` и помечает локальные сущности как `CONFLICT`; `requiresBootstrap` фиксируется отдельным флагом в `sync_state`
Причина:
- это дает первый рабочий end-to-end sync cycle без преждевременной фоновой оркестрации, retry-машины и conflict UI

### D-025
Статус: принято
Решение:
- app-level sync V1 строится вокруг одного `RunFullSyncUseCase` с простым ручным запуском из экрана настроек и порядком `bootstrap` либо `push -> changes`, в зависимости от `cursor/requiresBootstrap/outbox`
Причина:
- это дает реальную точку запуска sync из приложения без premature background orchestration и без сложной state machine

### D-026
Статус: принято
Решение:
- первый lifecycle-trigger sync V1 строится как foreground-trigger через `MainActivity.onStart()` и `AppSyncRunner`, с cooldown и защитой от параллельных запусков
Причина:
- это дает более естественный app-level запуск sync без premature background sync, WorkManager и сложной оркестрации жизненного цикла

### D-027
Статус: принято
Решение:
- sync-конфигурация клиента (`baseUrl`, `accessToken`) выносится из `sync_state` в отдельный DataStore-backed `SyncConfigStore`, который используется и ручным запуском, и foreground-trigger
Причина:
- это разделяет session/config и operational sync state, убирает debug-only ощущение от настройки синхронизации и готовит более чистую основу для будущего background sync

### D-028
Статус: принято
Решение:
- background sync foundation строится на одном unique periodic `SyncWorker` через WorkManager, с network constraint и использованием общего `RunFullSyncUseCase` и `SyncExecutionGate`
Причина:
- это дает безопасный фоновый фундамент без дублирования sync-логики, без premature scheduler-сложности и без конфликтов с ручным/foreground sync

### D-029
Статус: принято
Решение:
- client-side auth/session V1 строится вокруг `email + password`, backend auth endpoint-ов `register/login/refresh/me` и одного DataStore-backed `SyncSessionStore`, который хранит backend URL, токены и минимальный профиль пользователя
Причина:
- это переводит sync из debug-only режима в реальный пользовательский сценарий подключения к backend, не ломая уже готовый sync foundation

## План следующих шагов
1. Следующим шагом выбрать один из двух вариантов усиления UX вокруг уже готового sync foundation: conflict UI или polish auth/session UX.
2. После этого развивать реальную клиент-серверную интеграцию sync уже не только в foreground/manual/background foundation режиме, но и в более устойчивом пользовательском сценарии.
3. Затем уже итеративно усиливать retry, background orchestration и conflict-handling без пересборки базового контракта.
