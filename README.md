# Иви

Локальное Android-приложение для семейного учета событий по одному питомцу, собаке по имени Иви.

## Что это

«Иви» помогает вести понятную историю ухода за собакой без лишней сложности:

- карточка питомца с фото, именем и датой рождения;
- журнал событий с типами, статусами и датами контроля;
- история веса;
- локальные напоминания о важных датах;
- спокойный Compose UI на русском языке.

На текущем этапе Android-клиент уже выделен в отдельный Gradle root `android/`, а backend и синхронизация еще не реализованы.

## Что уже реализовано

- один питомец в рамках MVP;
- локальное хранение на `Room`;
- CRUD для питомца, типов событий, событий, веса и настроек напоминаний;
- фильтрация событий по статусам `ACTIVE`, `COMPLETED`, `ARCHIVED`;
- удаление события из режима редактирования;
- удаление типа события с правилом: без связанных событий удаляется, при наличии связанных событий деактивируется;
- выбор фото питомца через системный Android Photo Picker;
- локальное копирование фото во внутреннее хранилище приложения;
- локальные уведомления через `AlarmManager`, `BroadcastReceiver` и `NotificationChannel`;
- перепланирование напоминаний после изменений и после перезагрузки устройства;
- launcher-иконка приложения на основе фото Иви.

## Быстрый старт

```bash
./android/gradlew -p ./android :app:assembleDebug
```

Debug-сборка подписывается стандартным debug-ключом Android.

После установки debug-версии:

- база при первом запуске автоматически заполняется демо-данными;
- это поведение нужно для локальной разработки и проверки UX.

Для release-сборки приложение должно стартовать с пустой базой, без демо-данных.

## Release-подпись

В проекте уже подготовлена настройка release signing через файл `android/keystore.properties` в Android Gradle root.

Файл не хранится в git и должен быть создан локально.

Минимальные поля:

```properties
storeFile=/absolute/path/to/your-release.jks
storePassword=your_store_password
keyAlias=your_key_alias
keyPassword=your_key_password
```

Если `keyPassword` не указан, будет использован `storePassword`.

При запуске release-задач без настроенной подписи Gradle завершит сборку с ошибкой.

Пример release-сборки:

```bash
./android/gradlew -p ./android :app:assembleRelease
```

Важно: даже корректно подписанный APK при ручной установке может восприниматься Android как установка из неизвестного источника. Для «нормальной» пользовательской установки нужен релиз через Google Play, например через `Internal testing`.

## Архитектура

Android-часть теперь живет в `android/` как отдельный Gradle root и пока состоит из одного модуля `:app`, внутри которого соблюдается разделение на слои:

- `app/` — entrypoint, DI, навигация, тема;
- `core/` — общие UI-компоненты и утилиты;
- `data/` — Room, DAO, entity, repository-реализации и напоминания;
- `domain/` — модели и repository-контракты;
- `feature/` — экраны и `ViewModel` отдельных сценариев.

Основные пользовательские экраны:

- `home`
- `events`
- `eventedit`
- `eventtypes`
- `weight`
- `settings`
- `petedit`

## Данные и напоминания

- единственный источник истины на текущем этапе — локальная база `Room`;
- напоминания планируются только для событий со статусом `ACTIVE` и заполненной датой контроля;
- в debug при пустой базе создаются стартовые данные;
- в release автозаполнение отключено.

## Ограничения текущего этапа

Сейчас в проект не входят:

- backend;
- синхронизация между устройствами;
- несколько питомцев;
- web-версия;
- серверные push-уведомления.

Следующий крупный этап после стабилизации локального MVP — проектирование backend и sync-контракта.
